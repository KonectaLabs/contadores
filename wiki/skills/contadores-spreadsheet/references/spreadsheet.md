# Spreadsheet Reference

## Identity

- Spreadsheet URL/GID usually come from each funnel definition in
  `config/default-funnels.json`, `FUNNELS_CONFIG_PATH`, or `data/funnels.json`.
- Per-funnel fields: `sheet_url`, `sheet_gid`, `sheet_source_filter`.
- Backwards-compatible env keys for legacy Contadores scripts: `CONTADORES_SHEET_URL`
  and `CONTADORES_SHEET_GID`.
- Reader-script aliases: `GOOGLE_SHEET_URL` and `GOOGLE_SHEET_GID`.
- Reader script: [`src/tools/read_google_sheet.py`](/Users/fgoiriz/private/repos/contadores/src/tools/read_google_sheet.py)

`/api/runtime` reports `ready=true` when at least one enabled `campaign` funnel
has both `sheet_url` and `sheet_gid`.

For new niche funnels, spreadsheet config belongs in the funnel definition file:

- `config/default-funnels.json` for the portable seed
- `FUNNELS_CONFIG_PATH`, usually `data/funnels.json`, for per-server overrides
- per-funnel fields: `sheet_url`, `sheet_gid`, `sheet_source_filter`

For Client Lead Delivery, sheet config belongs in the Delivery source config
file rather than in funnel config:

- `CLIENT_LEAD_SOURCES_CONFIG_PATH`, usually `data/client-lead-sources.json`
- seed: `config/default-client-lead-sources.json`
- per-source fields: `sheet_url`, `sheet_gid`, `sheet_tab_name`,
  `sheet_poll_seconds`, `recipient_phone` or `recipients`, and
  `column_mapping`
- multiple campaign sheets for the same client can be declared through
  `sheets`, each with `id`, `label`, `sheet_url`, `sheet_gid`, and
  `sheet_tab_name`
- a `sheets` item can override `column_mapping` and `context_field_mapping` when
  that tab uses different column names. Omitted column mappings inherit from the
  parent source.

Multiple `recipients` are expanded into separate DB sources that share the same
sheet. This lets Codex create one client Delivery flow from one sheet and
several WhatsApp recipient numbers without using the UI.

The current Contadores env vars remain backwards-compatible defaults.
Click-to-WhatsApp ads do not need a sheet row. They use the WhatsApp webhook
`referral.source_id` and the funnel field `whatsapp_referral_source_ids`.
The approved Abogados prefilled proposal text can also create/reuse an
`abogados` funnel lead when no reply/referral route is usable.
If the webhook includes `contacts.profile.name`, that WhatsApp profile name is
stored as the lead `full_name` for WhatsApp-created leads and fills existing
phone-only leads. It does not replace names already provided by the sheet or an
operator.

## What The Spreadsheet Represents

This sheet contains inbound leads generated by Meta lead forms for the `contadores` offer.

It already captures:

- lead identity;
- lead creation timestamp;
- ad, ad set, campaign, and form metadata;
- platform information such as `fb` or `ig`;
- contact data such as email, full name, and phone number;
- a coarse operational flag for whether the lead was already contacted.

## Runtime Rule

There is no runtime mode switch for Contadores.

Enabled campaign funnels poll their configured sheet directly.
After the offer step, the app can send a recap inside `offer text` when
DSPy classifies the lead reply as a simple watched-video confirmation. That
state lives in the app database, not as a spreadsheet runtime switch.

Important nuance:

- product changes are expected to end on the real server by default;
- `localhost` is only the development, validation, git, push, and deploy workbench.

## Column Meaning

- `id`: Lead id from the source platform.
- `created_time`: When the lead was created. Keep the original timestamp string.
- `ad_id`: Ad identifier.
- `ad_name`: Ad display name.
- `adset_id`: Ad set identifier.
- `adset_name`: Ad set display name.
- `campaign_id`: Campaign identifier.
- `campaign_name`: Campaign display name.
- `form_id`: Lead form identifier.
- `form_name`: Lead form display name.
- `is_organic`: Whether the lead was organic.
- `platform`: Source platform, usually `fb` or `ig`.
- `email`: Lead email.
- `full_name`: Lead full name.
- `phone_number`: Lead phone number. Current values may include a `p:` prefix.
- `lead_status`: Source lead state from Meta.
- `is_contactado`: Coarse business flag indicating whether the lead was already contacted.

## How To Connect From Python

### Option 1: Public read

If the sheet is public, the simplest path is:

```bash
cd /Users/fgoiriz/private/repos/contadores
uv run python src/tools/read_google_sheet.py --as-records
```

The script:

- loads `.env` automatically;
- prefers `CONTADORES_SHEET_URL` and `CONTADORES_SHEET_GID`;
- falls back to `GOOGLE_SHEET_URL` and `GOOGLE_SHEET_GID`;
- tries public CSV export first.

### Option 2: Service account

Use this for private access or any write workflow.

Required steps:

1. Enable Google Sheets API in Google Cloud.
2. Create a service account.
3. Download the JSON key.
4. Share the sheet with that service account as `Editor`.
5. Set one of these env vars:
   - `CONTADORES_GOOGLE_SERVICE_ACCOUNT_FILE`
   - `GOOGLE_SERVICE_ACCOUNT_FILE`

Example:

```bash
CONTADORES_GOOGLE_SERVICE_ACCOUNT_FILE=/path/to/service-account.json \
uv run python src/tools/read_google_sheet.py --as-records
```

## Why The Sheet Is Useful

For this project, the sheet is useful because it is:

- visible to non-technical operators;
- easy to inspect and edit manually;
- enough for a poller-driven MVP;
- a practical source of truth for intake.

## Operational Rules

- The configured sheet is the intake source for enabled campaign funnels.
- Use explicit column names. Do not rely on positions.
- Deduplicate by the sheet row id.
- Do not mark `is_contactado` before the outbound send actually succeeds.
- Normalize phone numbers before using them downstream.
- Strip prefixes like `p:` before trying to send WhatsApp messages.

```text
wa_error
wa_lock_token

```

## Meaning Of Proposed Automation Columns

- `wa_status`: High-level state such as `pending`, `sent`, `waiting_reply`, `done`, `error`, `handoff`.
- `wa_sequence`: Which sequence the lead is following.
- `wa_step`: Current step in that sequence.
- `wa_last_outbound_at`: Timestamp of the last outbound WhatsApp message.
- `wa_last_inbound_at`: Timestamp of the last inbound reply.
- `wa_next_action_at`: When the poller should next act on this lead.
- `wa_message_id`: Provider message id of the latest outbound message.
- `wa_error`: Last send or sync error.
- `wa_lock_token`: Short-lived claim token used by the worker.
- `wa_lock_until`: Lock expiration timestamp.
- `wa_human_handoff`: Boolean flag to stop automation and let a human continue.
- `wa_notes`: Freeform operator notes.
