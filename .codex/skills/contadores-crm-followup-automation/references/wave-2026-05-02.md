# CRM Follow-Up Wave 2026-05-02

This is the historical runbook for the CRM wave executed on 2026-05-02 for the
`contadores` and `abogados` funnels.

## User Instructions That Shaped The Work

The user asked for a complete CRM analysis and lead-state plan:

- Analyze all CRM leads for Contadores and Abogados.
- Remove Venezuelans completely because they are bad leads.
- Remove Guido and Rodrigo from Workstation because they are already converted.
- Classify every other lead into a small number of action buckets.
- Pay special attention to leads in Manual that stopped answering; some are one
  message, two messages, or one call away from converting.
- Check message delivery status for every lead. If an outbound failed, treat it
  as our failure to answer, not as the lead ignoring us.
- Use the user's previous manual messages as copy inspiration, almost verbatim
  when they fit.
- Match the user's WhatsApp style: simple, direct, no opening question marks,
  often no accents, light personalization.
- Before sending, show the exact copies.
- Build a Python script to run on the server, preferably with hardcoded lead ids
  grouped by action, not a CSV.
- Send only the first message in each sequence. Later, after replies come in,
  another run can continue the sequence.
- Deploy, run live, then review every chat/message for errors before ending.

## Buckets Created

The wave used these action buckets:

- `repair_delivery`: requeue failed outbound messages so failed sends are not
  mistaken for no-reply leads.
- `answer_manual`: answer a manual lead where we owed a clear reply.
- `close_call`: warm/manual leads close enough to invite to a short call.
- `retomar_video`: leads that had received video/context and needed a simple
  "did you see it / want to continue" nudge.
- `reactivate_followup`: stale leads with some prior contact; use approved
  opener follow-up template.
- `cold_mass_followup`: cold leads that did not answer the initial opener; use
  approved opener follow-up template.

Hard exclusions:

- Venezuelans.
- Workstation clients.
- Closed, converted/legacy booked, archived.
- Leads where the current action was unsafe for the WhatsApp window.

## Script Design

The script is:

`src/scripts/contadores_followup_wave_20260502.py`

Design choices:

- Constants contain the grouped lead ids/message ids.
- Default mode is dry-run.
- `--live` is required to queue messages.
- `--scope contactable` includes priority, warm, reactivation, and cold
  follow-up buckets.
- It writes a preview CSV:
  `data/reports/crm-followup-wave-2026-05-02-preview.csv`.
- It writes a one-time ledger:
  `data/contadores/crm-followup-wave-2026-05-02-ledger.json`.
- The ledger prevents accidental duplicate live runs unless `--force` is used.
- It uses existing backend behavior instead of direct ad hoc sends:
  `queue_manual_message_for_lead`, `run_quick_action_for_lead`,
  `send_opener_followup`, and `ContadoresMessage.requeue_failed_delivery`.
- It blocks Workstation clients using `WorkstationClient.get_by_lead_id`.
- It blocks closed/converted/legacy booked/archived using the effective stage.
- It blocks Venezuelans. After live verification, the filter was tightened in
  commit `3dad6ee` to catch local Venezuelan mobile formats like `0412...`.

Commits:

## Lessons For Future Runs

- Never classify "no reply" before checking delivery status.
- Provider errors should be grouped by Meta code before deciding to retry.
- Local phone formats matter; do not rely only on normalized country prefixes.
- The best close action is not "send Calendly"; ask for day/time for a short
  15-minute call.
- After a mass wave, the next run should focus on new replies and near-close
  leads, not resending to the whole cold list every hour.
